# Architecture

## Guiding principles

- **Calculators are pure functions.** Take numbers in, return numbers out. No DB access, no printing. This makes them trivially testable and lets the CLI *and* the future web layer call them unchanged.
- **DB is read-only at runtime.** `seed_db.py` is the only writer. The rest of the app queries. Deleting `ksp.db` and re-seeding should always reproduce identical state.
- **The CLI is a view.** It formats data; it does not compute. Same rule for the web layer in Phase 8.
- **Mods, KSP2, and RSS are alternate seeds, not alternate code.** Schema stays put; swap the DB file.

## Module responsibilities

### `db.py`
- Single `sqlite3.Connection` factory with `row_factory = sqlite3.Row`
- Read-only helpers: `get_body(slug)`, `list_bodies(type=None)`, `get_antenna(name)`, `get_dsn(level)`
- Context manager for transactions (used only by seed scripts)

### `orbital.py`
Pure math. No DB, no I/O. Every function is a one-liner in terms of inputs and a physical law.
- `orbital_period(sma_m, mu) -> seconds`
- `vis_viva(sma_m, r_m, mu) -> m/s`
- `escape_velocity(r_m, mu) -> m/s`
- `hohmann_dv(r1_m, r2_m, mu) -> (dv1, dv2, total)`
- `surface_gravity(mu, radius_m) -> m/s²`
- `sync_orbit_radius(mu, rotation_period_s) -> m`
- `hill_sphere(sma_m, ecc, mass_ratio) -> m`

### `comms.py`
Pure. Takes body/antenna dicts from the DB layer, returns a report dict.
- `comm_range(p1_w, p2_w) -> m` — geometric mean of two antenna powers
- `min_sats_for_coverage(body_r, orbit_alt, min_elev_rad) -> int`
- `orbit_for_coverage(body_r, n_sats, min_elev_rad) -> m`
- `sat_separation(orbit_r, n_sats) -> m`
- `comm_network_report(body, n_sats, antenna, dsn_level, min_elev_deg) -> dict`

### `dv_map.py`
Pure. Operates on the Δv tree.
- `path_dv(from_slug, to_slug) -> list[Edge]` — tree walk via LCA
- `plan_trip(stops, margin_pct=5.0) -> TripPlan`
- `TripPlan.total_raw`, `TripPlan.total_planned` (with margin)

### `plans.py` *(Phase 6)*
The only module besides seeds that writes to the DB. Reads/writes the `plans` table. Saves a trip configuration (not the computed total — that is recomputed on load so stale chart values don't linger).

### `cli.py`
Typer app. Subcommands mirror calculators:
```
ksp body <slug>
ksp comms <body> --sats 3 --antenna RA-15 --dsn 2
ksp dv <from> <to> [--via <slug> --action orbit|land|flyby] [--margin 5]
ksp hohmann <from> <to>
ksp plan save|load|list|delete
```
Pretty output via Rich tables and boxed panels.

### `web/` *(Phase 8)*
FastAPI app that imports the same calculator modules and exposes them as JSON endpoints. Pydantic schemas double as request validation and OpenAPI docs.


## What is explicitly *not* here

- **No ORM.** `sqlite3` + raw SQL. Overhead isn't worth it at this scale.
- **No caching layer.** The DB is a few KB; every query hits disk; nothing to cache.
- **No async.** Calculators are CPU-bound and fast. FastAPI can call sync functions directly.
- **No migrations framework.** Schema changes = regenerate `ksp.db` from seeds. No user data lives in the DB until Phase 6, and the `plans` table is simple enough for hand-written `ALTER` if ever needed.
